# ADK 2.0 Migration Guide

## Overview

ADK Python 2.0.0 reached general availability on May 19, 2026, introducing
significant architectural changes from ADK 1.x ([ADK 2.0 announcement](https://adk.dev/2.0/)).
This guide documents the breaking changes and migration steps required when
upgrading from ADK 1.x to ADK 2.0.

## Architectural Shift

ADK 2.0 introduces the **Workflow Runtime**, transitioning ADK from a
hierarchical agent executor to a graph-based execution engine. Agents, Tools,
and Functions are now evaluated as individual nodes within a workflow graph.

## Breaking Changes

### ADK Python 1.x to 2.0 Incompatibilities

Several known incompatibilities exist when upgrading ADK 1.x projects to ADK 2.0.

#### 1. Event Schema Changes

ADK 2.0 introduces new fields `node_info` and `output` to the core Event
schema to track graph state and workflow outputs.

**Migration action:** Update your database schemas and downstream client
validators to expect and store the `node_info` and `output` fields on all
Event payloads. Ensure all reader applications are updated to handle the 2.0
format before writing 2.0 sessions to a shared database.

#### 2. Agent Execution: BaseAgent to BaseNode

In ADK 1.x, Agents were standalone executors. In ADK 2.0, the `BaseAgent`
class now subclasses `BaseNode`. Agents are evaluated as individual nodes
within the new Workflow Graph engine.

**Migration action:** Move custom execution logic out of `run()` overrides.
Instead, utilize the standardized `BeforeAgentCallback` and `AfterAgentCallback`
interfaces to safely inject custom logic into the execution lifecycle.

#### 3. Context and Callbacks: In-Place Mutation

Bypassing the framework to manually append events is no longer safe.

**Migration action:** Do not append events directly to the session, and do
not use `enqueue_event` directly. You must now explicitly yield the event
from within your node or agent so that the framework can manage its
persistence, routing, and streaming natively.

#### 4. Error Handling and Automatic Retries

The ADK 2.0 framework now automatically catches exceptions to enable
automatic retries, telemetry, and Human-in-the-Loop (HITL) pauses.

**Migration action:** Allow standard exceptions to propagate out of your
tools so the framework can evaluate them against your configured
`RetryConfig`, such as `RetryConfig(max_attempts=3)`. Never catch
`BaseException` unless you are explicitly re-raising the exception.

## Installing ADK Python 1.x

If you want to update ADK but are not yet ready to update to ADK 2.0, make
sure to specify an ADK version during installation or use the compatible
release `~=` operator as shown below.

### Installation Steps

1. **Enable a Python virtual environment:**

```bash
python3 -m venv .venv
```

2. **Activate the Python virtual environment:**

On macOS/Linux:

```bash
source .venv/bin/activate
```

On Windows (Command Prompt):

```cmd
.venv\Scripts\activate.bat
```

On Windows (PowerShell):

```powershell
.venv\Scripts\Activate.ps1
```

3. **Install the package using pip:**

```bash
pip install "google-adk~=1.0"
```

## Additional Incompatibilities

If you encounter additional ADK 1.0 to ADK 2.0 incompatibilities, report
them through the [ADK issue tracker](https://github.com/google-gemini/adk/issues).
